# 模块开发测试与提交流程

更新时间：2026-05-02 13:05 CST

## 固定执行原则

- 每次只聚焦一个模块，不并行推进多个未收口的大模块。
- 模块代码完成后，先做模块级验证，再做文档更新，再提交 GitHub。
- 只有当前模块测试通过并已推送远端，才进入下一个模块。

## 标准流程

1. 开发当前模块。
2. 运行模块级验证：
   - `scripts/verify_module.sh <module-key>`
3. 如果改动跨应用、跨前后端或影响公共服务：
   - `scripts/verify_module.sh full`
4. 更新文档：
   - `docs/开发记录.md`：记录完成内容、关键 API、限制和备注。
   - `docs/测试记录.md`：记录本次验证命令和结果。
   - `docs/Wiki草案.md`：如果页面、命令、测试基线或功能说明变化，需要同步。
5. 复查工作树：
   - `git status --short`
6. 提交代码：
   - `git add <相关文件>`
   - `git commit -m "<module summary>"`
7. 推送代码：
   - `git push origin main`
8. 推送成功后，切换到下一个模块继续开发。

## 模块键与验证命令

- `primary-approval`：一级总表审批闭环与多节点审批
- `cycle-task`：预算周期与任务分发
- `dashboard`：Dashboard 指标与下钻
- `template`：模板设计器深化
- `demands`：专题需求
- `masterdata`：主数据扩展
- `version-analysis`：版本分析高级能力
- `audit`：审计与提醒
- `full`：全量回归

## 文档更新最低要求

### `docs/开发记录.md`

- 写清本轮完成了什么。
- 写清有哪些业务规则变化。
- 写清新 API 或行为变化。

### `docs/测试记录.md`

- 写清执行了哪些命令。
- 写清测试数量、通过情况和失败修复情况。
- 写清本轮回归关注点。

### `docs/Wiki草案.md`

- 页面说明变化时同步。
- 运行命令变化时同步。
- 测试基线变化时同步。

## GitHub 合入门禁

- 本地模块级验证通过。
- 涉及公共逻辑时全量验证通过。
- 文档已同步。
- 工作树内容与提交说明一致。
- GitHub Actions CI 通过。

## 建议提交粒度

- 一个模块一个提交，最多一组紧密相关提交。
- 不把多个未完成模块混进同一次提交。
- 文档和代码一起提交，避免“代码到了，文档没到”。
